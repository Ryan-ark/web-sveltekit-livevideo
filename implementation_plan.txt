Implementation Plan: Live Video Viewing and 1:1 Video Calls

Date: 2026-03-18
Project: web-sveltekit-livevideo
Status: Review only, no implementation performed

1. Executive Summary

Yes, this project can support both:
- live video that only selected accounts can watch
- 1:1 video calls

The current repo already has a viable realtime foundation through Ably, but it is only being used for chat events today. The missing piece is not "websocket support" by itself. The missing piece is media transport and call/session orchestration.

Recommended architecture:
- Keep Ably for signaling, presence, invitation state, and room/call events
- Use WebRTC for camera/microphone media
- Add a TURN service for reliability
- For 1:1 calls: peer-to-peer WebRTC is appropriate
- For "one broadcaster, some accounts can watch": small viewer counts can work with direct WebRTC, but anything beyond a small audience should use an SFU/media platform such as LiveKit, Daily, Twilio Video, Agora, or Cloudflare Calls

Important constraint:
- This app is deployed with the Vercel adapter and is structured around stateless server routes. That is compatible with Ably-based signaling and WebRTC. It is not a good place to build and host your own in-process websocket media backend.

2. What Exists Today

2.1 Realtime layer

The project already uses Ably as the realtime transport layer.

Relevant files:
- src/lib/server/realtime/ably.ts
- src/routes/api/realtime/ably-token/+server.ts
- src/routes/chat/[conversationId]/+page.svelte

Current behavior:
- Server creates Ably token requests scoped to a single chat channel
- Client connects to Ably from the chat page
- Presence is used to show online/offline status
- Realtime events are used for:
  - message.created
  - typing.started
  - typing.stopped

Evidence in code:
- src/lib/server/realtime/ably.ts:3-8 grants publish/subscribe/presence/history on a private chat channel
- src/lib/server/realtime/ably.ts:39-55 creates token requests for `private:chat:{conversationId}`
- src/routes/api/realtime/ably-token/+server.ts:10-27 only issues a token after conversation membership is verified
- src/routes/chat/[conversationId]/+page.svelte:162-231 creates an Ably realtime client, attaches the channel, enters presence, and subscribes to chat events

Assessment:
- Good foundation for signaling
- Not yet a media solution
- Realtime auth pattern is already correct in principle: authorize first on the server, then mint scoped realtime access

3. Feasibility Review

3.1 1:1 video call

Feasibility: High

Reason:
- You already have authenticated users
- You already have direct conversations between two members
- You already have a secure realtime signaling path using Ably

What is still needed:
- WebRTC peer connection setup
- SDP offer/answer exchange
- ICE candidate exchange
- local/remote media stream management
- call lifecycle state
- TURN/STUN configuration

Conclusion:
- 1:1 video calling is realistic in this codebase without major architectural rework

3.2 Live video that only some accounts can watch

Feasibility: High, but design choice matters

There are two different interpretations:

Option A: "One broadcaster, a few allowed viewers"
- Feasible with the current stack plus WebRTC signaling
- Works for small numbers only
- Broadcaster uploads one stream per viewer if using pure peer-to-peer mesh
- CPU and uplink cost grows linearly with viewer count

Option B: "One broadcaster, many allowed viewers"
- Feasible, but not with chat-style Ably events alone
- Requires an SFU/media backend
- This is the correct path if you expect moderate concurrency, reliability requirements, screen sharing growth, recording, or future group calls

Conclusion:
- If this feature is not strictly limited to a very small audience, plan for an SFU-backed design from the start

4. Architectural Recommendation

Recommended target design:

4.1 Use Ably only for signaling and control plane

Ably should carry:
- call invitation
- ringing / accepted / declined / ended
- SDP offer / answer
- ICE candidates
- mute / camera on-off / screen-share state
- viewer join / leave events
- call presence and occupancy

Ably should not carry:
- raw audio/video media

Reason:
- Media over websocket-style messaging is the wrong transport
- WebRTC is designed for this problem

4.2 Use WebRTC for media plane

For 1:1 calls:
- `RTCPeerConnection`
- `getUserMedia`
- STUN + TURN ICE servers

For gated live viewing:
- Small audience path: broadcaster establishes separate peer connections to each approved viewer
- Scalable path: use SFU provider and keep Ably for app-level state only

4.3 Model calls/streams separately from chat messages

Do not overload the current chat message model with call state.

Recommended new concepts:
- call_session
- call_participant
- stream_room or broadcast_room
- room_viewer / room_member
- call_event or audit table

Reason:
- Calls are ephemeral stateful sessions
- Messages are durable textual history
- The operational and authorization rules are different

5. Key Gaps in the Current Repo

5.1 No media/session schema

Today there is no database model for:
- active call records
- call invitations
- participant device state
- stream viewers
- room ownership or allowed viewers
- call history / ended reason / duration

Impact:
- You can prototype without persistence, but production behavior becomes hard to recover, debug, and moderate

5.2 Realtime scope is chat-only

Today the realtime helper only knows how to mint tokens for chat channels:
- src/lib/server/realtime/ably.ts:39-55
- src/routes/api/realtime/ably-token/+server.ts:10-27

Impact:
- You need a second signaling namespace for calls/streams
- Ideally token capabilities should be more granular than "chat publish/history/presence"

5.3 Conversation model is only direct

Today the enum is:
- src/lib/server/db/schema/chat.ts:13-15 => `direct`

Impact:
- Fine for 1:1 calls
- Not enough for a room or broadcast audience feature

5.4 No TURN/STUN configuration

Impact:
- WebRTC will fail for a meaningful portion of users on NAT-restricted or enterprise networks if TURN is not available

5.5 No UI state machine for calls

Today the chat page supports:
- presence
- message feed
- typing indicator

It does not support:
- incoming call prompt
- ringing state
- permission errors
- reconnect / retry
- device selection
- media preview
- ended / failed / busy states

6. Recommended Delivery Strategy

The cleanest path is to deliver in phases.

Phase 1: 1:1 call foundation
Goal:
- Add reliable 1:1 audio/video calls between members of an existing direct conversation

Why first:
- The data model already has direct conversations
- The auth rule is easy: only conversation members can call or answer
- The UI insertion point already exists in the chat conversation page

Scope:
- Start / accept / decline / end call
- audio + video
- mute / camera toggle
- presence-based availability
- TURN-backed ICE
- minimal call history persistence

Phase 2: gated live viewing for small audience
Goal:
- Allow an authenticated broadcaster to start a live session that only approved accounts can watch

Scope:
- room creation
- host role
- approved viewer membership
- join / leave events
- host stream + viewer playback
- hard audience cap if using peer-to-peer broadcast

Phase 3: scalable live rooms or future group calling
Goal:
- Replace or augment peer-to-peer broadcast with an SFU when concurrency or reliability requires it

Scope:
- provider integration
- room tokens
- server-issued access grants
- optional recording and moderation

7. Detailed Implementation Plan

7.1 Domain and data model design

Add a new media feature module, separate from chat.

Suggested file area:
- src/lib/server/features/media/

Suggested schema additions:
- media_call_sessions
  - id
  - conversationId nullable
  - roomId nullable
  - mode (`direct_call`, `broadcast`)
  - status (`ringing`, `active`, `ended`, `missed`, `declined`, `failed`)
  - initiatedByUserId
  - startedAt
  - endedAt
  - endedReason
  - createdAt
  - updatedAt

- media_call_participants
  - callSessionId
  - userId
  - role (`caller`, `callee`, `host`, `viewer`)
  - joinStatus (`invited`, `joined`, `declined`, `missed`, `left`, `kicked`)
  - joinedAt
  - leftAt

- media_rooms
  - id
  - slug or public key if needed
  - createdByUserId
  - title
  - visibility (`private`, `project`, `role_limited`)
  - status (`idle`, `live`, `ended`)
  - createdAt
  - updatedAt

- media_room_members
  - roomId
  - userId
  - accessRole (`host`, `producer`, `viewer`)
  - approvedByUserId nullable
  - createdAt

- media_call_events (optional but recommended)
  - id
  - callSessionId
  - actorUserId nullable
  - type
  - payload json/text
  - createdAt

Notes:
- For 1:1 only, you can anchor calls to `chatConversations.id`
- For gated live viewing, use a separate room model; do not force this into the direct chat schema

7.2 Authorization rules

Recommended initial rules:

For direct 1:1 calls:
- caller must be a member of the conversation
- callee must be the other member of the conversation
- only active participants can publish signaling events for that call

For live rooms:
- only host or producer can start the live session
- only approved members can join
- only host can publish upstream media in the first version

Implementation shape:
- add media-specific guard/service methods similar to `chatService.assertConversationMembership`
- do server-side authorization before issuing any realtime token or call join payload

7.3 Realtime channel strategy

Do not reuse `private:chat:{conversationId}` for everything.

Recommended channels:
- `private:chat:{conversationId}` for messages and typing only
- `private:call:{callSessionId}` for direct call signaling
- `private:room:{roomId}` for broadcast room control and viewer presence

Suggested event names for direct calls:
- `call.invited`
- `call.accepted`
- `call.declined`
- `call.ended`
- `webrtc.offer`
- `webrtc.answer`
- `webrtc.ice-candidate`
- `participant.media-state.updated`

Suggested event names for live rooms:
- `room.live.started`
- `room.live.ended`
- `viewer.joined`
- `viewer.left`
- `webrtc.offer`
- `webrtc.answer`
- `webrtc.ice-candidate`

Implementation note:
- extend `src/lib/server/realtime/ably.ts` into a generic realtime helper for chat plus media
- create distinct token issuance methods instead of one chat-specific method

7.4 Token issuance and capability scoping

Current token route:
- src/routes/api/realtime/ably-token/+server.ts

Recommended change later:
- keep this route or add a new media token route
- validate target resource type and id
- return a token scoped only to the relevant call/room channel

Security notes:
- keep channels private
- never let the client choose arbitrary channel names without server validation
- use short token TTLs for call/room access

7.5 WebRTC implementation shape

Client-side responsibilities:
- request camera/mic permission
- create `RTCPeerConnection`
- attach local tracks
- create and apply offers/answers
- exchange ICE candidates through Ably
- render remote tracks in video elements
- clean up devices and peer connection on leave/end

Suggested client structure:
- src/lib/features/media/client/webrtc.ts
- src/lib/features/media/client/call-controller.svelte.ts or equivalent Svelte state module
- reusable Svelte components for:
  - local preview
  - remote video tile
  - call controls
  - incoming call modal
  - device permission banner

Server-side responsibilities:
- create call session records
- authorize join/start actions
- issue scoped realtime credentials
- record lifecycle transitions

7.9 Presence and concurrency rules

Define these rules explicitly before implementation:
- if a user is already in an active direct call, can they receive another call?
- can a user watch a live room while in a direct call?
- does starting a second call end the first one, fail, or mark busy?
- is only one active live room allowed per host?

Recommended initial behavior:
- one active direct call per user
- one active broadcast per host
- if callee is busy, emit `call.declined` with busy reason

7.10 Error and edge-case handling

Plan for these cases explicitly:
- user denies camera/mic permission
- network interruption after call established
- caller leaves tab during ringing
- callee opens two tabs
- duplicate Ably subscriptions after route navigation
- media device unplugged mid-call
- ICE connection fails and needs restart
- host ends live room while viewers are connected

7.11 Testing plan

Unit tests:
- authorization rules for call/room access
- input schema validation
- call state transitions
- room membership rules

Integration tests:
- token endpoint denies unauthorized access
- creating call session requires valid conversation membership
- room viewer access is restricted to approved users

Manual verification checklist:
- two users can start and answer a 1:1 call
- busy state behaves correctly
- mute/camera toggles update peer state
- browser refresh ends or reconnects according to spec
- unauthorized user cannot join call or live room
- approved viewer can watch host stream
- unapproved viewer is blocked

Reality check:
- end-to-end browser media testing is difficult to cover fully with the current test setup alone
- manual multi-browser verification will be required

8. Recommendation on Broadcast Strategy

If your intended live feature means:

"One host and up to 2-4 viewers"
- Peer-to-peer broadcast is acceptable for a first release
- Faster to implement
- Lower infrastructure complexity

"One host and more than a handful of viewers"
- Skip peer-to-peer broadcast and adopt an SFU provider
- This avoids redoing the architecture shortly after launch

My recommendation:
- Build 1:1 direct calling first with WebRTC + Ably signaling
- For gated live viewing, decide now whether the audience is truly small
- If there is any chance this becomes a team room, training stream, demo stream, or multi-viewer feature, plan SFU-backed rooms instead of mesh

9. Risks and Tradeoffs

Risk: Trying to send media through the existing realtime layer
- Outcome: high latency, poor quality, expensive, wrong abstraction

Risk: Reusing chat channels for signaling
- Outcome: event collisions, authorization ambiguity, harder debugging

Risk: Launching without TURN
- Outcome: call setup failures for real users

Risk: Forcing live rooms into the current `direct` conversation schema
- Outcome: awkward permissions model and migration debt

Risk: Starting with mesh broadcast if audience can grow
- Outcome: host upload bottleneck and inevitable rewrite

10. Recommended Build Order

1. Create media domain schema and service layer
2. Add media-specific authorization guards
3. Extend Ably helper for call/room channels and token issuance
4. Implement direct call lifecycle APIs
5. Implement 1:1 WebRTC client on the chat conversation page
6. Add TURN/STUN config and environment validation
7. Add manual test checklist and state transition tests
8. Add live room schema and access model
9. Implement gated live viewing for small audiences or integrate SFU provider

11. Final Recommendation

This is possible in the current project without major structural rework.

The strongest path for this repo is:
- keep Ably as the signaling/authenticated realtime layer
- add WebRTC for actual audio/video
- use the existing direct conversation membership model for 1:1 calls
- create a separate room model for "some accounts can watch"
- decide early whether broadcast is small-peer mesh or SFU-backed

If implementation starts later, the first milestone should be a direct 1:1 video call inside the existing chat conversation page. That fits the current codebase cleanly and exercises the right abstractions before adding private live rooms.
